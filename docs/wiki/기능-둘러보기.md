# 기능 둘러보기

**Git Tab에서 할 수 있는 모든 기능을 알아봅시다. 각 기능 옆에는 "언제 쓰나?"를 붙였습니다.**

---

## 🆕 저장소 복제 (Clone)

### 뭐하는 기능?
GitHub, GitLab 같은 원격 저장소의 URL을 입력하면, 그 저장소를 통째로 내 PC로 복제합니다. 원격의 모든 커밋 이력, 브랜치, 파일이 그대로 복사됩니다.

### 사용법

**방법 1: 메인 창에서**
1. Git Tab 메인 창의 **"복제"** 버튼을 클릭합니다.
2. 저장소 URL을 입력합니다 (예: `https://github.com/user/repo.git`).
3. 저장할 폴더를 선택합니다.
4. "복제" 버튼을 클릭하면 다운로드가 시작됩니다.

**방법 2: Windows 탐색기에서**
1. 탐색기에서 저장소를 만들 폴더를 우클릭합니다.
2. **"Git Tab"** → **"복제"** 메뉴를 선택합니다.
3. 저장소 URL을 입력합니다.
4. "복제" 버튼을 클릭합니다.

### 언제 쓰나?
- GitHub/GitLab에 있는 오픈소스 프로젝트를 내 컴퓨터에 내려받을 때
- 팀 저장소를 처음 받을 때
- 다른 사람의 코드를 로컬에서 실행해보고 싶을 때

## 🎨 컬러풀한 커밋 그래프

### 뭐하는 기능?
화면 중앙의 커밋 그래프가 브랜치마다 다른 색으로 표현됩니다. 각 커밋 옆에는 작성자 아바타, 커밋 메시지, 변경된 파일 개수(막대 그래프)가 표시됩니다.

### 언제 쓰나?
- 여러 브랜치에서 작업할 때 시각적으로 누가 뭘 했는지 한눈에 파악
- 병합 이력을 추적할 때
- 팀원들의 작업 현황을 빠르게 이해할 때

## 📝 변경사항 스테이징 & 커밋

### 기본 흐름

1. **파일 목록 보기**  
   우측 "변경사항" 섹션에서 수정된 모든 파일이 표시됩니다.

2. **파일 선택하기**  
   커밋하고 싶은 파일 앞의 ✓ 체크박스를 클릭해 스테이징합니다.

3. **메시지 입력** (커밋 메시지 도우미 활용)  
   아래 텍스트 박스에 "뭘 했는지" 간단히 설명합니다.  
   메시지 앞에는 **커밋 종류 드롭다운**이 있습니다 (Conventional Commits 표준):
   ```
   종류: [feat] [fix] [docs] [style] [refactor] [test] [chore] 등
   예) feat: 회원가입 폼 유효성 검사 추가
   예) fix: 로그인 페이지 다크 테마 버그 수정
   예) docs: README 업데이트
   ```
   드롭다운에서 적절한 종류를 선택하면 메시지 앞에 자동으로 붙습니다.

4. **커밋하기**  
   "커밋" 버튼을 클릭합니다.

### 추가 옵션

- **부분 스테이징 (Partial Staging):** 파일 전체가 아니라, 그 파일의 일부만 커밋하고 싶을 때.  
  변경 파일 옆의 **"±" 버튼**을 클릭하면, 그 파일의 바뀐 부분(헝크, hunk) 하나하나를 선택해 스테이징할 수 있습니다. (git add -p와 동일)
  
- **파일 되돌리기:** 수정 전 상태로 복원할 수 있습니다. 파일을 우클릭 → "되돌리기(Discard)" 선택.

- **Amend(수정):** 방금 커밋한 메시지나 파일을 수정하고 싶을 때. 체크박스에서 "Amend" 옵션을 켜고 다시 커밋하면 마지막 커밋이 수정됩니다.

### 언제 쓰나?
- 매일 변경사항을 기록할 때
- 기능별로 따로 커밋하고 싶을 때
- 잘못 커밋한 메시지를 고칠 때

## 🌿 브랜치 & 태그

### 브랜치 생성
1. 우측 "브랜치" 탭을 클릭합니다.
2. "+ 새 브랜치" 버튼을 클릭합니다.
3. 브랜치 이름을 입력합니다.
   ```
   예) feature/로그인
   예) bugfix/회원가입-오류
   예) release/v1.0.0
   ```
4. "생성" 버튼을 클릭하면 새 브랜치가 만들어집니다.

### 브랜치 전환 (체크아웃)
브랜치 목록에서 전환하고 싶은 브랜치를 우클릭 → "체크아웃(Checkout)" 선택. 현재 브랜치가 바뀝니다.

### 브랜치 이름 변경
브랜치를 우클릭 → "이름 변경(Rename)" 선택. 새 이름을 입력합니다.

### 브랜치 삭제
브랜치를 우클릭 → "삭제(Delete)" 선택. 현재 브랜치는 삭제할 수 없습니다.

### 브랜치 병합 (Merge)
1. 병합하고 싶은 대상 브랜치로 먼저 체크아웃합니다.  
   예) `main`에 `feature/로그인` 병합하고 싶다면, 먼저 `main`으로 전환.
2. 병합할 브랜치(예: `feature/로그인`)를 우클릭 → "병합(Merge)" 선택.
3. 충돌이 없으면 완료. 충돌이 있으면 [문제 해결](문제-해결) 페이지를 확인하세요.

### 리베이스 (Rebase)
고급 기능입니다. 두 브랜치를 병합하는 다른 방법. 브랜치를 우클릭 → "리베이스(Rebase)" 선택. 커밋 이력이 더 깔끔해집니다.

> **처음에는 병합만 사용해도 충분합니다.**

### 태그 생성
1. 커밋 그래프에서 태그를 붙이고 싶은 커밋을 우클릭합니다.
2. "태그 생성(Create Tag)" 선택.
3. 태그 이름을 입력합니다.
   ```
   예) v1.0.0
   예) v2.1.1-beta
   ```
4. "생성" 버튼을 클릭합니다.

### 언제 쓰나?
- **브랜치:** 새 기능 개발, 버그 수정, 실험할 때
- **병합:** 기능 완성 후 메인에 반영할 때
- **태그:** 버전 출시할 때

## 🔀 대화형 리베이스 (Advanced)

커밋들을 재정렬하거나, 여러 커밋을 하나로 합치거나(squash), 특정 커밋을 지울 수 있는 고급 기능입니다.

### 기본 사용법
1. 브랜치를 우클릭 → "대화형 리베이스" 또는 "Interactive Rebase" 선택.
2. 기준이 될 커밋(보통 `main`)을 선택합니다.
3. 각 커밋 옆의 액션을 선택합니다:
   - **pick:** 이 커밋을 그대로 사용
   - **squash:** 이전 커밋과 합치기
   - **reorder:** 순서 변경
   - **drop:** 이 커밋 삭제

### 언제 쓰나?
- Pull Request 전에 커밋을 깔끔하게 정리할 때
- "작은 커밋 3개를 1개로 합치고 싶을 때"

> **주의:** 고급 기능이므로 처음 사용할 때는 테스트용 브랜치에서 연습하세요.

## 📦 스태시 (Stash)

### 저장하기
1. 우측 메뉴에서 "스태시 저장(Save Stash)" 버튼을 찾습니다.
2. 스태시에 이름을 붙입니다. (선택사항)
   ```
   예) "로그인 폼 작업 중"
   ```
3. "저장" 버튼을 클릭합니다.

작업 중인 파일들이 임시 보관되고, 작업 폴더가 깨끗해집니다.

### 꺼내기
1. "스태시 목록" 또는 "Stash List"를 확인합니다.
2. 복구하고 싶은 스태시를 우클릭 → "적용(Apply)" 또는 "꺼내기(Pop)" 선택.

두 옵션의 차이:
- **Apply:** 스태시를 적용하되, 스태시는 목록에 남습니다.
- **Pop:** 스태시를 적용하고 목록에서 지웁니다.

### 언제 쓰나?
- 작업 중인 코드를 건드리지 않고 긴급 버그를 고칠 때
- 커밋 없이 브랜치를 전환해야 할 때

## 🔍 Blame

각 줄의 코드가 누가 언제 작성했는지 확인합니다.

### 사용법
1. 커밋 그래프에서 커밋을 우클릭합니다.
2. "Blame" 또는 "이력 보기" 선택.
3. 각 줄 옆에 작성자, 커밋 메시지, 날짜가 표시됩니다.

### 언제 쓰나?
- "이 버그는 어디서 온 거야?"라고 추적할 때
- 팀원에게 "이 부분 왜 이렇게 했어?"라고 물어봐야 할 때

## ⚠️ 충돌 해결 (Conflict Resolution)

두 브랜치를 병합할 때 같은 파일의 같은 부분이 다르게 수정됐다면 충돌이 발생합니다.

### 충돌 파일 찾기
우측 "변경사항" 섹션에 빨간색으로 표시됩니다.

### 충돌 해결 방법

**방법 1: Git Tab의 시각적 3-way 병합 편집기**
1. 충돌 파일을 우클릭합니다.
2. "충돌 해결(Resolve Conflict)"을 선택합니다.
3. **시각적 3-way 병합 편집기**가 열립니다:
   - **왼쪽 (Base):** 원본 코드
   - **중앙 (결과):** 최종 결과 (편집 가능)
   - **오른쪽 (Theirs / HEAD):** 상대의 변경사항
   - 아래: "내 것으로", "상대 것으로", "양쪽 유지" 등 빠른 채우기 버튼
4. 필요한 부분을 선택하거나 직접 수정해 결과를 완성합니다.
5. 저장하면 파일이 자동으로 스테이징됩니다.

**방법 2: 직접 수정**
1. 파일을 텍스트 에디터로 열어봅니다.
2. 충돌 표시를 찾습니다:
   ```
   <<<<<<< HEAD
   내 버전
   =======
   다른 버전
   >>>>>>> 브랜치이름
   ```
3. 필요한 부분만 남기고 충돌 표시를 삭제합니다.
4. Git Tab에서 "해결됨(Resolved)" 선택.

### 중단/계속
- **중단:** 병합을 취소하고 처음 상태로 돌아갑니다. "Abort Merge" 옵션 선택.
- **계속:** 충돌 해결 후 병합을 완료합니다. "완료(Finish)" 또는 "커밋" 버튼 클릭.

### 언제 쓰나?
- 팀으로 작업할 때 피할 수 없는 상황

## 🌐 원격 (Remote)

### Push (서버로 올리기)
1. 우상단 **"Push"** 버튼을 클릭합니다.
2. 인증 정보를 입력해야 하면 사용자 이름 + 토큰 창이 뜹니다.
3. "확인" 또는 "Push" 버튼을 클릭합니다.

### Pull (서버에서 받기)
1. 우상단 **"Pull"** 버튼을 클릭합니다.
2. 팀원들의 새 커밋이 다운로드됩니다.

### Fetch (최신 정보만 확인)
1. 우상단 **"Fetch"** 버튼을 클릭합니다.
2. 서버 정보가 업데이트되지만, 파일은 변경되지 않습니다.

### Ahead / Behind 표시
현재 브랜치 옆에 숫자가 표시됩니다:
- **Ahead 3:** 로컬에 서버보다 3개 커밋 더 있음 (Push 필요)
- **Behind 2:** 서버에 로컬보다 2개 커밋 더 있음 (Pull 필요)

### GitHub / GitLab 연동

**PR(Pull Request) / MR(Merge Request) 만들기**
- 현재 브랜치로 빠르게 PR/MR 페이지를 열 수 있습니다.
- 우상단 **"PR 만들기"** 버튼을 클릭하면, 웹 브라우저에서 GitHub/GitLab의 PR/MR 생성 페이지가 열립니다.

**웹에서 열기**
- 현재 저장소를 웹 브라우저에서 보고 싶을 때.
- 우상단 **"웹에서 열기"** 또는 저장소 설정에서 **"저장소 홈"** 버튼을 클릭하면, GitHub/GitLab 저장소 홈 페이지가 열립니다.

### 언제 쓰나?
- GitHub/GitLab과 코드를 동기화할 때
- 팀원과 협업할 때
- 다른 팀원의 코드를 Pull Request로 리뷰할 때

## 🔧 .gitignore 생성기

Git이 추적하지 않아야 할 파일들(예: 비밀번호, 임시 폴더, 컴파일 산출물)을 자동으로 설정합니다.

### 사용법
1. 메뉴에서 ".gitignore 생성" 또는 "Generate .gitignore" 선택.
2. 프로젝트 유형을 선택합니다:
   ```
   예) Python, Node.js, C#, Java, Android 등
   ```
3. Git Tab이 자동으로 필요한 규칙들을 추가합니다.

### 언제 쓰나?
- 새 저장소를 만들 때
- 프로젝트를 Git으로 처음 설정할 때

## 📋 파일 히스토리 (File History)

### 뭐하는 기능?
특정 파일을 건드린 모든 커밋들을 한눈에 봅니다. 그 파일의 변경 이력을 추적할 때 유용합니다.

### 사용법
1. 변경사항 섹션에서 파일을 우클릭합니다.
2. **"파일 히스토리 보기"** 또는 **"View File History"** 선택.
3. 그 파일을 건드린 모든 커밋이 시간순으로 나열됩니다.
4. 각 커밋을 클릭하면 그 커밋에서의 diff(변경사항)를 볼 수 있습니다.

### 언제 쓰나?
- "이 버그는 언제부터 있었지?"라고 추적할 때
- 특정 파일의 변경 이력을 상세히 알고 싶을 때
- "이 기능은 누가 언제 추가했어?"라고 확인할 때

## 🔀 비교 (Compare)

### 뭐하는 기능?
두 브랜치, 두 태그, 또는 두 커밋 사이의 차이를 한눈에 봅니다. 어떤 파일들이 바뀌었는지, 각 파일에서 뭐가 달라졌는지 볼 수 있습니다.

### 사용법
1. 상단 툴바의 **"비교"** 또는 **"Compare"** 버튼을 클릭합니다.
2. 비교할 대상 (From/To)을 선택합니다:
   - 브랜치 선택 (예: `main` vs `feature/로그인`)
   - 태그 선택 (예: `v1.0.0` vs `v1.1.0`)
   - 커밋 선택
3. "비교" 버튼을 클릭하면:
   - 변경된 파일 목록이 표시됩니다.
   - 각 파일을 클릭하면 diff를 볼 수 있습니다.

### 언제 쓰나?
- Pull Request 전에 내 변경사항을 미리 확인할 때
- "v1.0.0부터 v1.1.0까지 뭐가 바뀌었지?"라고 확인할 때
- 팀원의 브랜치와 메인의 차이를 검토할 때

## 🔍 내용 검색 (Pickaxe)

### 뭐하는 기능?
코드 전체를 훑어서, 특정 문자열이 **추가되거나 삭제된** 커밋들을 찾아줍니다. 정규식(regex)도 지원합니다.

### 사용법
1. 상단 툴바의 **"검색"** 버튼을 클릭합니다.
2. 찾고 싶은 문자열을 입력합니다:
   ```
   예) "const MAX_RETRY = "
   예) "function setupAuth"
   예) 정규식: "^import.*module"
   ```
3. "검색" 버튼을 클릭합니다.
4. 해당 문자열이 추가되거나 삭제된 커밋들만 필터링됩니다.
5. 각 커밋을 클릭하면 그 부분의 diff를 확인할 수 있습니다.

### 언제 쓰나?
- "이 함수는 언제 제거됐어?"라고 추적할 때
- 버그가 있는 코드 라인이 어느 커밋에서 추가됐는지 찾을 때
- 특정 의존성 임포트가 어디서 추가됐는지 찾을 때

## ⌨️ 명령 팔레트 (Command Palette)

### 뭐하는 기능?
키보드 단축키 **`Ctrl+P`** 하나로 모든 Git Tab 액션을 검색해 빠르게 실행합니다.

### 사용법
1. **`Ctrl+P`**를 누릅니다.
2. 명령 팔레트 창이 열립니다.
3. 하고 싶은 기능을 입력해 검색합니다:
   ```
   예) "commit" → 커밋 대화상자 열기
   예) "push" → Push 실행
   예) "clone" → 저장소 복제
   예) "rebase" → 리베이스
   ```
4. 엔터를 누르거나 마우스로 클릭해 실행합니다.

### 언제 쓰나?
- 메뉴를 클릭하는 것보다 빠르게 기능을 실행할 때
- 기억나지 않는 기능의 정확한 이름을 검색할 때
- 키보드만으로 작업 흐름을 진행할 때

## 🔄 작업 기록 & 되돌리기 (Reflog)

### 뭐하는 기능?
최근에 HEAD가 가리킨 모든 지점(커밋)들의 목록입니다. 실수로 브랜치를 삭제하거나 리셋했을 때, 예전 상태로 복원할 수 있습니다.

### 사용법
1. 상단 툴바의 **"↩"** (되돌리기/Reflog) 버튼을 클릭합니다.
2. 최근 작업 목록이 표시됩니다:
   ```
   예) "HEAD@{0}: commit 메시지"
   예) "HEAD@{1}: checkout main"
   예) "HEAD@{2}: reset 이전 커밋"
   ```
3. 돌아가고 싶은 지점을 선택합니다.
4. "복원" 또는 "Checkout" 버튼을 클릭하면 그 시점으로 HEAD가 이동합니다.

### 언제 쓰나?
- 실수로 잘못 커밋한 직후 "마지막 작업 취소"할 때
- 삭제한 브랜치를 복구해야 할 때
- 복잡한 rebase 중에 문제가 생겼을 때

## ⏸️ 작업 취소 (Cancel Operations)

### 뭐하는 기능?
Fetch, Pull, Push, Clone 같은 장시간 걸리는 작업을 중간에 멈출 수 있습니다.

### 사용법
1. Fetch, Pull, Push, Clone 중에 진행 대화상자 또는 상태바에 **"취소" / "Cancel"** 버튼이 나타납니다.
2. 그 버튼을 클릭하면 작업이 중단됩니다.
3. 작업 내용은 부분적으로만 적용될 수 있으므로 주의하세요.

### 언제 쓰나?
- 네트워크가 느릴 때 큰 저장소 clone을 취소할 때
- 잘못된 브랜치를 Pull 중에 발견했을 때
- 의도하지 않은 Push를 중단할 때

## 🐛 크래시 리포트 (Crash Report) - 옵트인

### 뭐하는 기능?
Git Tab이 오류로 종료될 때, 오류 정보를 **로컬 디스크**에 저장해서 나중에 개발팀에 보고할 수 있습니다. 외부로 데이터를 전송하지 않습니다.

### 사용법
1. 우상단 **⚙ 설정** 아이콘을 클릭합니다.
2. **"진단"** 또는 **"Diagnostics"** 섹션을 찾습니다.
3. **"크래시 리포트 수집 활성화"** 체크박스를 켭니다.
4. 이제부터 앱이 종료될 때 오류 정보가 자동으로 저장됩니다.

### 리포트 위치
```
C:\Users\<사용자명>\AppData\Local\GitTab\CrashReports\
```

### 언제 쓰나?
- 개발팀에 버그를 보고할 때, 크래시 리포트를 첨부하면 더 빠르게 해결됩니다.
- 앱의 안정성을 높이는 데 도움을 줄 때.
